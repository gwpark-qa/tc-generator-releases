### 시스템 관련
- 레어도 : 유니크/슈퍼에픽/에픽/레어/언커먼/커먼
- 뽑기 : 쿠키 및 아티팩트를 얻는 수단
- 타워패스 : 유료로 구입하는 배틀패스형 상품
- 기차 레벨 : 게임 플레이로 얻는 경험치로 올릴 수 있는 레벨
- 탑 레벨 : 별도의 미션을 클리어해야 올릴 수 있는 레벨
- 기차 로비 : 게임 메인 로비화면

### 쿠키 관련
- 쿠키 : 인게임 플레이어블 캐릭터 및 NPC를 지칭함. 게임 세계관 내 인간에 해당하는 존재
- 영혼석 : 쿠키 성급 성장용 재화. 20영혼석 = 1쿠키
- 아티팩트 : 각 쿠키의 고유한 강화요소.
- 장비 : 무기, 모자, 상의, 하의 장비가 존재. 잠재력 적용 가능.
- 잠재력 : 장비 아이템에 특수 재화로 적용할 수 있는 추가 능력치
- 속성 : 총 6개의 속성, 불/물/바람/대지/어둠/빛. 각 쿠키는 하나의 속성을 보유
- 역할군 : 대미지 딜러, 스트라이커, 서포터.
- 타입 : 베기, 타격, 마법, 사격, 지원

### 재화 관련
- 코인 : 쿠키/장비 등의 강화에 사용되는 인게임 재화
- 크리스탈 : 뽑기/입장권 구입 등에 사용되는 현금 재화
- 하트젤리 : 스토리모드, 성장던전 입장에 사용되는 재화
- 무지개큐브 : 스킨 뽑기에 사용되는 재화

### 콘텐츠 관련
- 레이드 : 멀티 환경에서 4인이 협동해 보스를 잡는 콘텐츠
- 레이드 챌린지 모드 : 조금 더 어려운 난이도의 레이드
- 어비스 레이드 : 최상위 콘텐츠, 4인 협동은 동일하나 랜덤매칭이 아닌 사전 매칭으로 진행
- 어비스 던전 : 어비스 레이드 파생 콘텐츠. 설탕유리 조각과 시즈나이트 원석 파밍
- 성장 던전 : 각 던전에 해당하는 성장재화 파밍
- 탑의 균열 : 장비 및 장비 강화 재화를 파밍하는 웨이브 클리어 형식의 콘텐츠
- 아카데미 전술시험 : 8개의 쿠키를 이용한 자동 전투 콘텐츠
- 스토리 모드 : 정해진 스테이지를 따라가는 스토리 모드.
- 어드벤처 모드 : 8개의 스테이지로 진행되는 로그라이크 콘텐츠
- 그림자 토벌 : 지정 조합으로 보스를 공략하는 콘텐츠

### 인게임 관련
- 속성표식 : 스트라이커가 특수 스킬/궁극기로 적 공격 시 적에게 부여되는 표식
- 속성강타 : 속성표식의 게이지를 채우면 표식이 있는 대상에게 가해지는 강력한 공격

## 인게임 시스템 정의

### 코어 루프
- 목적: 스테이지 입장 → 주어진 목표 클리어 → 승리
- 승리 조건: [도착지점 도달, 몬스터 모두 처치, 보스 클리어] (콘텐츠별 상이)
- 패배 조건: [모든 쿠키 사망, 제한시간 초과]
- 제한시간: 콘텐츠별 상이. UI 좌측 상단 표시

### 파티 구성 (콘텐츠별)
| 구분 | 구성 | 해당 콘텐츠 |
|------|------|-------------|
| 싱글 기본 | 쿠키 3종 × 1명 | 대부분의 싱글 콘텐츠 |
| 멀티 기본 | 쿠키 1종 × 4명 | 레이드, 성장던전 등 |
| 멀티 특수 | 쿠키 3종 × 2명 | 스토리 모드, 탑의 균열, 어비스 모드 |

### 조작 체계

#### 이동
- 좌측 가상 조이스틱

#### 우측 버튼 (4개)
| 버튼 | 위치 | 터치 | 홀드 |
|------|------|------|------|
| 공격 | 우측 중앙(가장 큰 비중) | 기본 공격 | 차지 공격 (일부 쿠키, 쿠키별 홀드시간 상이, 별도 게이지 표시) |
| 대시 | 공격 좌측 | 이동기 사용 | - |
| 특수스킬 | 대시 상단 | 특수 스킬 사용 | - |
| 궁극기 | 특수스킬 우측 | 궁극기 사용 (무적 판정 + 연출) | - |

#### 방향 지정
- 공격/스킬 버튼 홀드 → 원하는 방향으로 슬라이드 = 해당 방향으로 사용
- 미지정 시: 자동 조준

#### 태그 (쿠키 교체)
- 위치: 화면 우측 상단 쿠키 아이콘
- 조작: 쿠키 아이콘 터치 = 해당 쿠키로 태그
- 태그 쿨타임: 약 1.5초
- 태그궁: 쿠키 아이콘 좌측의 궁극기 아이콘 터치 = 해당 쿠키로 태그 + 즉시 궁극기 사용
- 태그궁 조건: 해당 쿠키의 궁극기가 쿨타임이 아닐 것

### 스킬 시스템
- 쿨타임 적용 대상: 이동기, 특수스킬, 궁극기
- 쿨타임 표시: 사용 직후 해당 버튼 위 남은 쿨타임 숫자 표시
- 궁극기 특수 판정: 시전 시 무적 판정 + 궁극기 연출

### 이동기 (대시) 종류
| 유형 | 설명 |
|------|------|
| 기본 대시 | 짧은 거리 순간 이동 (대부분의 쿠키) |
| 점멸 | 짧은 거리 순간이동 |
| 은신 + 이속 증가 | 은신 상태 진입 + 이동속도 버프 |
| 이중 대시 | 대시 2회 연속 사용 가능 |

- 대시 중 무적 판정: ❌ 없음. 대미지 정상 적용

### 피격 시스템
- 일부 공격 피격 시: 행동 캔슬 + 스턴
- 슈퍼아머: 별도 존재하지 않음
- 무적 활용: 궁극기 시전 시 무적 판정으로 대체

### 역할군 & 딜 흐름

#### 역할군 정의
| 역할군 | 역할 | 핵심 기능 |
|--------|------|-----------|
| 대미지 딜러 | 대미지 특화 | 순수 딜링 |
| 스트라이커 | 속성 표식 부여 | 속성강타 유발 |
| 서포터 | 아군 지원 | 힐/보호막/버프/디버프 |

#### 딜 흐름 (기본 사이클)
스트라이커(표식 부여) → 서포터(버프/디버프) → 대미지 딜러(딜링)

#### 속성 강타 메카닉
1. 스트라이커가 특수스킬/궁극기로 적 공격 → 적에게 속성 표식 부여
2. 표식이 부여된 적을 아군이 공격 → 속성 강타 게이지 충전
3. 게이지 만충 → 게이지 초기화 + 대상에게 큰 대미지
- 속성 표식은 스트라이커 쿠키의 속성을 따름
- 동일 속성으로 공격 시 게이지 충전량 증가

### 사망 & 부활 시스템

#### 싱글
- 모든 쿠키 사망 = 패배

#### 멀티
- 쿠키 사망 → 유령 상태 전환
- 부활 방법 1 (아군 부활):
  - 유령 상태 플레이어에게 접근 → 공격 버튼이 "살려주기" 버튼으로 변경
  - 살려주기 버튼 꾹 누르기 → 해당 유령의 부활 게이지 충전 → 부활
  - 살려주기 중 피격 시: 캐스팅 취소됨, 단 충전된 게이지는 유지 (초기화 안 됨)
- 부활 방법 2 (자동 부활):
  - 유령 상태에서 일정 시간 경과 → 자동 부활
- 전멸 조건: 모든 쿠키가 동시에 유령 상태 = 게임 실패
- 부활 횟수: 콘텐츠별 제한 있음

# TC 작성 규칙

## 1. 기본 규칙
- 기획서 내 '의도' 또는 '방향성'은 TC로 제작하지 않습니다. 기획서 상 "클라이언트, 서버, UX 기능" 위주로 테스트 케이스로 작성합니다.
- 하나의 기획서는 하나의 시트로 통합합니다. (여러개의 시트로 나누지 않기)
- 유사한 케이스는 통합하여 작성 (예: 개별 스탯 표시 → "기본 스탯이 출력된다"로 통합)
  - 통합 판단 기준: 동일 UI/동일 동작/동일 기대결과 구조를 가진 케이스. 조건·대상·결과 중 하나라도 달라지면 분리한다.
- 기획서 내 편의상 영어로 작성된 표현은 TC 작성 시 적절한 한국어로 변환하여 작성
  - 예시: enter_transit → 입장 전환, idle_state → 대기 상태, reward_popup → 보상 팝업
- TC 단일 기대결과 원칙: 하나의 TC에는 하나의 기대결과만 작성한다. 두 개 이상의 기대결과가 포함되는 경우, 각각 별도의 TC로 분리하여 작성한다.
  - 예시(X): "상단 스토리/도전 모드 전환 토글에서 스토리 탭 선택 시 스토리 리스트 화면이 출력되고, 도전 탭 선택 시 도전 로비 화면으로 전환된다"
  - 예시(O): 아래와 같이 두 개의 TC로 분리하여 작성
    - TC 1: "상단 스토리/도전 모드 전환 토글에서 스토리 탭 선택 시, 스토리 리스트가 출력된다."
    - TC 2: "각 도전 리스트의 도전 배너 선택 시, 도전 로비 화면으로 전환된다."

## 2. 자료 분석 방법
  - 텍스트 자료와 이미지 자료가 동일 내용을 포함할 수 있습니다.
  - 이미지에서 먼저 1차적으로 내용과 맥락을 파악하고, 첨부한 텍스트가 있는 경우, 텍스트와 비교분석하여 정확한 수치/조건/스펙을 확인합니다.
  - 두 자료를 교차 검증하여, 텍스트에만 있는 내용과 이미지에만 보이는 내용을 모두 TC에 반영하세요.
  - 이미지에서 보이지만 텍스트에 언급되지 않은 UI 요소(버튼, 아이콘, 상태 표시 등)도 TC로 작성하세요.

## 3. TC 순서 규칙
- UI 플로우에 첨부한 문서를 참고하여, 유저 경험 상 순서대로 TC 작성 (사용자가 이용하기 편한 흐름 기반) / 예시: 콘텐츠 해금 -> 튜토리얼 -> 메인 로비 → 모드리스트 → 콘텐츠 메인화면 -> 콘텐츠 내 각 기능

- 카테고리 계층 구조 정의: 모든 TC는 아래 4단계 카테고리 체계에 따라 분류하여 작성한다. 각 단계의 역할이 다르므로, 단순히 main/sub1만 채우지 말고 sub2, sub3까지 적극적으로 세분화하여 작성한다.
  - **main (최상위 카테고리)**: 기획서 내 가장 큰 단위의 기능/콘텐츠 영역. 예) "메인 로비", "도전 로비", "상점", "이벤트", "튜토리얼"
  - **sub1 (main의 하위 기능)**: main 카테고리에 속한 주요 화면 또는 기능 단위. main에서 진입할 수 있는 하위 화면이나 주요 기능 그룹을 의미. 예) "아카데미 코스터", "보상 수령", "친구 초대", "상품 목록"
  - **sub2 (sub1의 하위 기능)**: sub1 내부의 세부 화면, 탭, 영역, 또는 기능 그룹. sub1을 더 잘게 나눈 기능 단위. 예) "상단 탭 영역", "보상 리스트", "입력 영역", "상품 상세 정보", "진행도 표시"
  - **sub3 (sub2의 하위 기능)**: 유저 상호작용에 의해 일어나는 가장 작은 기능/사건 단위. 대부분 "팝업", "전환", "제거", "등록", "출력", "활성화", "터치 동작", "입력 처리" 등 실질적인 유저 행동의 결과 또는 미세한 UI 동작을 의미. 예) "확인 팝업", "화면 전환", "레드닷 제거", "닉네임 등록", "버튼 활성화", "보상 지급 연출"
  - 작성 원칙:
    - 가능한 한 sub3까지 작성하여 TC를 세밀하게 구조화한다. 단순한 기능이라도 "팝업", "전환", "등록" 등의 상호작용 단위가 존재하면 sub3로 분리한다.
    - 더 이상 분화할 수 없는 단위에 도달했을 때만 해당 sub 컬럼을 빈 문자열("")로 둔다.
    - 같은 sub2 내에서 여러 유저 상호작용(팝업 출력, 버튼 동작, 입력 처리, 화면 전환 등)이 발생하는 경우, 반드시 각각을 sub3로 분리하여 작성한다.

- 카테고리 통합 원칙: 하나의 카테고리는 해당 카테고리에 속하는 모든 기능이 한 번에 완결되도록 작성한다.
  - 상위 카테고리(main) 내 버튼이나 진입점을 통해 이동하는 하위 화면의 기능은, 별도의 main 카테고리로 분리하지 않고 해당 상위 카테고리 안에 sub 카테고리를 만들어 포함시킨다.
  - 즉, 특정 화면에서 버튼을 눌러 진입하는 하위 화면의 모든 기능(화면 전환, UI 요소, 동작, 결과 등)은 해당 버튼이 위치한 카테고리의 하위(sub1, sub2, sub3 등)에 묶어서 작성한다.
  - 예시 1: "도전 로비"라는 main 카테고리 안에 "아카데미 코스터" 버튼이 있을 경우, "아카데미 코스터"를 별도의 main 카테고리로 분리하지 않는다. 대신 main: "도전 로비" > sub1: "아카데미 코스터"로 구성하고, 아카데미 코스터 화면 전환 후 적용되는 UI 요소, 기능, 동작 등을 모두 해당 sub1 카테고리 내에서 작성한다.
  - 예시 2 (sub3까지 분화): main: "이벤트" > sub1: "친구 초대" > sub2: "초대 입력 영역" > sub3: "닉네임 등록" / sub3: "확인 팝업" / sub3: "전송 완료 메시지"와 같이, 하나의 sub2 안에서 발생하는 각 유저 상호작용을 sub3로 세분화한다.
  - 이를 통해 테스트 수행 시 하나의 카테고리를 열면 관련 하위 기능까지 연속으로 검증할 수 있도록 한다.

- 화면 이동 케이스 배치 규칙: 다른 화면으로 이동하는 기능의 TC는 해당 카테고리의 마지막에 작성한다.
  - 예시: "초대장 팝업"이라는 sub1 카테고리가 있을 때, "방문하기 버튼 터치 시, 아카데미 전술시험 화면으로 이동된다."라는 케이스는 해당 카테고리의 마지막에 작성한다.
- 예외사항(edge case) 배치 규칙: 주요 기능이 아닌 예외사항은 해당 카테고리 내에서 가장 마���막에 위치시킨다.
  - 예외사항 판단 기준: 정상 플로우가 아닌 비정상 조건(앱 종료/재실행, 네트워크 단절, 조건 미달, 오류 상황 등)에서 발생하는 동작.
  - 배치 기준: 해당 예외 케이스가 속한 카테고리 중 가장 작은(하위) 카테고리를 기준으로 그 카테고리의 끝에 작성한다.
    - main 카테고리 전체에 해당하는 예외사항 → main 카테고리의 맨 끝에 배치
    - sub1 카테고리에 해당하는 예외사항 → 해당 sub1 카테고리의 맨 끝에 배치
    - sub2 카테고리에 해당하는 예외사항 → 해당 sub2 카테고리의 맨 끝에 배치
  - 예시: "게임을 껐다 켰을 때 튜토리얼이 이어서 재생된다"는 '튜토리얼' 기능의 예외적인 상황이므로, 튜토리얼이 속한 카테고리의 마지막에 배치
- 튜토리얼 기획서가 첨부된 경우, 튜토리얼 케이스를 먼저 작성한 후 메인 기획서 내용을 작성한다.

## 4. 상세도 규칙
- 핵심 원칙: 기획서의 모든 기능이 TC로 충분히 반영되었는가 / 작성된 케이스 100건 이상 권고
- 커버리지가 100% 이지만 작성된 케이스가 100건이 안되는 경우에는 무리하게 100건을 채우지 않고 작성을 종료
- 조건, 대상, 효과 등 어떤 기능에 대한 사전 조건과 결과를 하나의 케이스에 구체적으로 명시
  - 예시(X): "스택을 폭발시킬 수 있다"
  - 예시(O): "차지 공격 사용 시, 축적된 스택이 폭발되어 발동 대상에게 화염 피해가 적용된다"
- 시각적 정보 없이 알 수 없는 케이스는 글로 풀어서 작성하거나, 문구로 설명하는 것이 불가능한 경우 해당 부분을 제거하고 작성한다.
  - 스크린샷이나 이미지를 첨부하지 않는 이상 독자가 이해할 수 없는 표현은 사용하지 않는다.
  - 예시(X): "인게임 튜토리얼(1)에서 기본 전투/인게임 화면 안내가 출력된다."
  - 예시(O): "인게임 튜토리얼 중 '기본 전투 조작 안내' 단계에서 이동, 공격, 회피 버튼에 대한 안내 UI가 출력된다."
  - 번호, 기호 등으로만 구분된 항목은 해당 항목의 내용을 텍스트로 구체적으로 서술하거나, 내용을 특정할 수 없는 경우 해당 참조 부분을 제거한다.
- 다만 아래 항목들은 빠짐없이 포함:
  - 각 UI의 모든 버튼 동작
  - 뒤로가기 플로우 (버튼/디바이스 뒤로가기) - 해당 기능 TC인 경우만
  - 알림 시스템 (레드닷/그린닷 출력/제거 조건)
  - 효과음, 애니메이션, 연출

## 5. 제외 규칙
- 비기능 케이스(해상도/노치/폰트 등)은 작성하지 않는다.
- 아웃게임 관련 케이스는 작성하지 않는다.
  - 예: 각 콘텐츠에서 Play 버튼을 누른 후 로딩화면을 거쳐 인게임으로 진입하는 경우, "Play 버튼을 터치 시, 로딩화면으로 진입한다." 이후의 인게임 기능만 작성한다.

## 6. Priority 기준
### 판단 원칙
- Priority는 "해당 기능에 결함이 있을 때 서비스에 미치는 영향도"를 기준으로 판단한다.
- Critical은 전체 TC의 약 10~15% 이내로 엄격하게 제한한다. 확신이 서지 않으면 Major로 분류한다.
- "핵심 기능이다"라는 이유만으로 Critical을 부여하지 않는다. 해당 기능의 결함이 게임 진행 불가, 데이터 손실, 재화 오류 중 하나에 해당하는지 반드시 확인한다.

### Critical (전체의 10~15%)
서비스 치명적 결함 — 해당 결함이 발생하면 유저가 게임을 정상적으로 이용할 수 없거나, 복구 불가능한 피해가 발생하는 경우
- 인게임 요소: 쿠키 조작, 보스/몬스터 동작
- 데이터 손실 오류 : 게임의 결과가 결과화면에 정상적으로 반영되지 않음 등
- 핵심 진입 완전 차단: 인게임 진입 불가, 게임결과 화면 진입 불가 등
- 각 하위 카테고리의 메인 기능
- 예시:
  - "상점에서 재화를 사용하여 아이템 구매 시, 재화는 차감되지만 아이템이 지급되지 않는다" → Critical
  - "메인 로비에서 특정 모드 진입 버튼 터치 시, 앱이 크래시된다" → Critical
  - 친구초대 이벤트의 테스트 케이스의 경우 "친구초대 팝업창" -> "친구초대 팝업 창 내 확인 버튼 터치 시, 입력한 닉네임의 친구에게 친구초대가 발송된다"(확인 버튼을 눌렀을 때 기능하지 않는다면 친구초대라는 "핵심기능"이 동작하지 않으므로 Critical)

### Major (전체의 40~50%)
주요 기능 결함 — 기능이 의도대로 동작하지 않지만, 게임 진행 자체는 가능하거나 우회할 수 있는 경우
- 핵심 기능의 동작 확인: 주요 콘텐츠(전투, 육성, 매칭 등)가 기획 의도대로 작동하는지
- 주요 진입/이탈 플로우: 각 화면 간 전환, 뒤로가기 동작 등 (완전 차단이 아닌 경우)
- 보상 지급/차감 처리: 보상이 올바르게 지급·표시되는지 (데이터 손실 수준이 아닌 경우)
- 주요 UI 요소: 핵심 정보 표시(레벨, 재화량, 진행도 등)의 정확성
- 상태 변경 및 조건부 동작: 조건에 따른 활성화/비활성화, 잠금/해제 등
- 알림 시스템: 레드닷/그린닷의 출력 및 제거 조건
- 예시:
  - "이벤트 메인 화면에서 보상 수령 버튼 터치 시, 보상이 지급된다" → Major
  - "스테이지 클리어 후 결과 화면에서 획득 보상 목록이 출력된다" → Major
  - "조건 미달 시, 해당 버튼이 비활성화 상태로 표시된다" → Major

### Normal (전체의 35~50%)
경미한 결함 — 기능 자체는 정상이나, 부가적 요소에 이슈가 있는 경우
- 효과음 및 배경음: 각 상황별 사운드 재생 여부
- 애니메이션 및 연출: 화면 전환 연출, 획득 연출, 이펙트 등
- 보조 UI 요소: 툴팁, 도움말, 부가 정보 표시
- 텍스트 및 표기: 문구 정확성, 줄바꿈, 말줄임 처리 등
- 예시:
  - "보상 획득 시, 연출 애니메이션이 재생된다" → Normal
  - "버튼 터치 시, 효과음이 출력된다" → Normal
  - "장비 상세 팝업에서 툴팁이 표시된다" → Normal

## 7. 출력 형식
- JSON 배열 형식으로 출력
- 컬럼: no, priority, main, sub1, sub2, sub3, expectationResult
- 컬럼 정의:
  - `no` (숫자, 필수): 1부터 시작하는 연속된 일련번호. 배열 순서와 일치해야 한다.
  - `priority` (문자열, 필수): "Critical" | "Major" | "Normal" 중 하나.
  - `main` (문자열, 필수): 최상위 카테고리명.
  - `sub1` (문자열, 선택): 하위 카테고리. 해당 없으면 빈 문자열 "".
  - `sub2` (문자열, 선택): 하위 카테고리. 해당 없으면 빈 문자열 "".
  - `sub3` (문자열, 선택): 하위 카테고리. 해당 없으면 빈 문자열 "".
  - `expectationResult` (문자열, 필수): 단일 기대결과. 문체 규칙(8번) 준수.

## 8. 문체 규칙
- 조건절 어미 규칙: TC 작성 시 조건절의 어미 "~하면"은 "~시,"로 변환하여 작성한다.
  - 예시(X): "콘텐츠 해금 조건을 달성하면, 해당 콘텐츠가 활성화된다"
  - 예시(O): "콘텐츠 해금 조건 달성 시, 해당 콘텐츠가 활성화된다"
- '정상' 표현 금지: "정상 작동된다", "정상 이동된다", "정상적으로", "정상적인", "정상" 등의 표현은 사용하지 않는다. 기능 동작 자체만 서술한다.
  - 예시(X): "보상이 정상 지급된다"
  - 예시(O): "보상이 지급된다"
- 수동형 표현 사용: 능동형보다 수동형 표현을 우선 사용한다.
  - 예시(X): "방문하기 버튼을 터치하면 아카데미 전술시험 화면으로 이동한다"
  - 예시(O): "방문하기 버튼 터치 시, 아카데미 전술시험 화면으로 이동된다"
