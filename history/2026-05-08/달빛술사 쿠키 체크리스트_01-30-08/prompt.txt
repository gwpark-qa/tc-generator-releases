# TC 작성 규칙

## 1. 기본 규칙
- 기획서 내 '의도' 또는 '방향성'은 TC로 제작하지 않음. 기획서 상 "클라이언트, 서버 기능"에 대해서 테스트 케이스로 작성
- 하나의 기획서는 하나의 시트로 통합
- 유사한 케이스는 통합하여 작성 (예: 개별 스탯 표시 → "기본 스탯이 정상 출력된다"로 통합)
- 기획서 내 편의상 영어로 작성된 표현은 TC 작성 시 적절한 한국어로 변환하여 작성
  - 예시: enter_transit → 입장 전환, idle_state → 대기 상태, reward_popup → 보상 팝업
  - 코드명이나 고유 명칭(예: API명, 시스템 고유 ID 등)은 영어 그대로 유지 가능

## 2. TC 순서 규칙
- UI 흐름 순서대로 TC 작성 (사용자가 이용하기 편한 흐름 기반)
- 예시: 메인 로비 → 모드리스트 → 이벤트 메인 → 플레이 → 결과 화면 → 팝업
- **카테고리 통합 원칙**: 하나의 카테고리는 해당 카테고리에 속하는 모든 기능이 한 번에 완결되도록 작성한다.
  - 상위 카테고리(main) 내 버튼이나 진입점을 통해 이동하는 하위 화면의 기능은, 별도의 main 카테고리로 분리하지 않고 해당 상위 카테고리 안에 sub 카테고리를 만들어 포함시킨다.
  - 즉, 특정 화면에서 버튼을 눌러 진입하는 하위 화면의 모든 기능(화면 전환, UI 요소, 동작, 결과 등)은 해당 버튼이 위치한 카테고리의 하위(sub1, sub2, sub3 등)에 묶어서 작성한다.
  - 예시: "도전 로비"라는 main 카테고리 안에 "아카데미 코스터" 버튼이 있을 경우, "아카데미 코스터"를 별도의 main 카테고리로 분리하지 않는다. 대신 main: "도전 로비" > sub1: "아카데미 코스터"로 구성하고, 아카데미 코스터 화면 전환 후 적용되는 UI 요소, 기능, 동작 등을 모두 해당 sub1 카테고리 내에서 작성한다.
  - 이를 통해 테스트 수행 시 하나의 카테고리를 열면 관련 하위 기능까지 연속으로 검증할 수 있도록 한다.
- **예외사항(edge case) 배치 규칙**: 주요 기능이 아닌 예외사항은 해당 카테고리 내에서 가장 마지막에 위치시킨다.
  - 배치 기준: 해당 예외 케이스가 속한 카테고리 중 **가장 작은(하위) 카테고리**를 기준으로 그 카테고리의 끝에 작성한다.
    - main 카테고리 전체에 해당하는 예외사항 → main 카테고리의 맨 끝에 배치
    - sub1 카테고리에 해당하는 예외사항 → 해당 sub1 카테고리의 맨 끝에 배치
    - sub2 카테고리에 해당하는 예외사항 → 해당 sub2 카테고리의 맨 끝에 배치
  - 예시: "게임을 껐다 켰을 때 튜토리얼이 이어서 재생된다"는 '튜토리얼' 기능의 예외적인 상황이므로, 튜토리얼이 속한 카테고리의 마지막에 배치

## 3. 상세도 규칙 (100건 이상 권고)
- **핵심 원칙**: 기획서의 모든 기능이 TC로 충분히 반영되었는가
- 내용이 적은 기획서는 100건 미만이어도 무방
- **발동 조건, 발동 대상, 발동 효과를 구체적으로 명시**
  - 예시(X): "스택을 폭발시킬 수 있다"
  - 예시(O): "차지 공격 사용 시, 축적된 스택을 폭발하여 발동 대상에게 화염 피해를 준다"
- **시각적 정보 없이 알 수 없는 케이스는 글로 풀어서 작성하거나, 여의치 않으면 해당 부분을 제거하고 작성한다.**
  - 스크린샷이나 이미지를 첨부하지 않는 이상 독자가 이해할 수 없는 표현은 사용하지 않는다.
  - 예시(X): "인게임 튜토리얼(1)에서 기본 전투/인게임 화면 안내가 출력된다."
  - 예시(O): "인게임 튜토리얼 중 '기본 전투 조작 안내' 단계에서 이동, 공격, 회피 버튼에 대한 안내 UI가 출력된다."
  - 번호, 기호 등으로만 구분된 항목은 해당 항목의 내용을 텍스트로 구체적으로 서술하거나, 내용을 특정할 수 없는 경우 해당 참조 부분을 제거한다.
- 다만 아래 항목들은 빠짐없이 포함:
  - 각 UI의 모든 버튼 동작
  - 뒤로가기 플로우 (버튼/디바이스 뒤로가기) - 해당 기능 TC인 경우만
  - 알림 시스템 (레드닷/그린닷 출력/제거 조건)
  - 효과음, 애니메이션, 연출

## 4. 제외 규칙
- 특정 기능 TC인 경우: 비기능 케이스(해상도/노치/폰트) 생략
- **스탯 수치 표현 규칙**: 기본스탯 설정에서 '공/생/방 : 상/중/하'와 같이 정확한 수치로 표현되지 않은 경우 TC 작성을 생략한다. 단, 값이 n 또는 x 등 미지수로 기록된 경우에는 TC로 작성한다.
  - 예시(생략): "공격력: 상, 체력: 중, 방어력: 하" → TC 작성하지 않음
  - 예시(작성): "공격력: n, 쿨타임: x초" → TC로 작성
- **체크리스트 네거티브 케이스 제외**: 체크리스트의 경우, 기획서에 적힌 스펙과 반대되는 네거티브 케이스(negative case)는 작성하지 않는다. 체크리스트는 테스트 케이스에 비해 가벼운 항목이므로, 명시된 스펙의 정상 동작 확인만 작성한다.
  - 예시: "특수기술 적중 시, 쿠키에게 받는 피해 증가 부여"라는 케이스가 있을 때, "기본공격 적중 시에는 쿠키에게 받는 피해 증가 미부여"와 같은 반대 급부의 케이스는 작성하지 않음
- **체크리스트 마이크로 밸런스 제외**: 체크리스트의 경우, 공격 활성 시간, 후딜 시간 등 마이크로한 밸런스 수치는 TC로 작성하지 않는다.
  - 예시: "공격 활성 시간 0.5초", "후딜 시간 1.2초" 등의 세부 프레임/시간 밸런스 항목은 제외
- **전투 사이클 제외**: 전투 사이클(combat cycle) 부분은 TC로 작성하지 않는다.

## 5. Priority 기준
- Critical: 핵심 기능, 진입/이탈, 보상 지급, 스킬 발동
- Major: 주요 UI 요소, 상태 변경, 조건부 동작
- Normal: 효과음, 애니메이션, 보조 기능

## 6. 출력 형식
- JSON 배열 형식으로 출력
- 컬럼: no, priority, main, sub1, sub2, sub3, expectationResult
