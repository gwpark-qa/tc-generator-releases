# 작업 원칙

## 1. 보존 원칙 (가장 중요)
- 기존 TC의 **no(번호)는 절대 변경하지 마세요**. 같은 기능이면 no를 유지.
- 변경 계획서에서 명시되지 않은 TC는 **한 글자도 수정하지 마세요**. 임의 개선·문구 다듬기 금지.
- 카테고리(main / sub1~3) 구조는 가능한 한 유지. 새 기획서가 명백히 재분류를 요구하지 않는 한.

## 2. 변경 판단 기준
- **수정 대상**: 새 기획서에서 같은 기능의 동작·수치·조건이 달라진 경우
- **추가 대상**: 새 기획서에서 새로 등장한 기능 중 기존 TC에 대응 항목이 없는 경우
- **제거 대상**: 새 기획서에서 명시적으로 삭제·축소된 기능에 해당하는 기존 TC
- **유지(변경 없음)**: 새 기획서에 언급은 있으나 기존 TC와 차이가 없거나, 새 기획서에 언급조차 없는 경우

## 3. 변경 근거
- 모든 수정/추가/제거는 **새 기획서의 구체적 부분을 근거**로 해야 합니다.
- 추측·일반화로 변경 금지. ("아마 바뀌었을 것이다" 같은 판단 X)
- 같은 동작이지만 표현만 다른 경우 → 변경 X (단순 문구 차이는 무시)
- 변경 계획서 단계에서는 "이유" 필드에 새 기획서의 어느 부분에 근거하는지 명시.

## 4. 표현·톤 일관성
- 기존 TC의 어투·양식과 동일하게 작성. 새 TC가 기존 TC 사이에 자연스럽게 섞여야 함.
- **발동 조건 / 발동 대상 / 발동 효과**를 구체적으로 명시.
  - 예시(X): "스택을 폭발시킬 수 있다"
  - 예시(O): "차지 공격 사용 시, 축적된 스택을 폭발하여 발동 대상에게 화염 피해를 준다"

## 5. Priority 기준
- Critical: 핵심 기능, 진입/이탈, 보상 지급, 스킬 발동
- Major: 주요 UI 요소, 상태 변경, 조건부 동작
- Normal: 효과음, 애니메이션, 보조 기능
- 변경된 TC의 Priority는 **기본적으로 기존 그대로 유지**. 변경된 기능의 중요도가 명백히 달라진 경우에만 조정.

## 6. 추가 TC 번호 부여
- 기존과 완전히 다른 신규 기능인 경우, 기존 TC의 가장 큰 no 다음 번호부터 **순차 부여**.
- 원 기능에서 변경되거나, 세분화 되거나, 기타 이유로 추가 케이스를 적어야 하는경우, 기존 케이스 사이에 행을 추가하여 케이스를 넣을 수 있음. 

## 7. 누락 방지
- 새 기획서에 등장한 기능이 기존 TC에 없는데 명시적으로 "제거되지 않은" 경우 → **추가 대상**으로 분류.
- 단순히 기존 TC에 없다고 무시하지 마세요.

# 단계별 흐름 안내
이 프롬프트는 TC 수정 흐름의 두 단계에서 호출됩니다:
1. **변경 계획서 단계** — 어떤 TC를 어떻게 변경할지 마크다운으로 분석/계획
2. **TC 수정 실행 단계** — 계획서대로 최종 TC JSON 배열 출력

각 단계의 출력 형식은 시스템이 추가로 명확히 지시합니다. 그 지시를 정확히 따라주세요.

# 출력 요청
위 원칙에 따라 기존 TC를 새 기획서에 맞춰 정확하게 갱신해주세요. 기존 TC를 가능한 한 보존하면서, 새 기획서의 변경 내용만 정확히 반영합니다.
